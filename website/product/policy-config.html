Policy Configuration

Configure the server-side policy file that controls which IdPs are accepted, how usernames are mapped, and what security checks are enforced.

Policy File

The PAM module reads its configuration from /etc/prmana/policy.yaml. This file controls all server-side authentication behavior. Create the directory and copy the example:

sudo mkdir -p /etc/prmana sudo cp /usr/share/prmana/examples/policy.yaml /etc/prmana/policy.yaml sudo chmod 600 /etc/prmana/policy.yaml sudo chown root:root /etc/prmana/policy.yaml
Copy

The policy file is automatically reloaded when modified -- no daemon restart required. The PAM module detects changes via stat(2) on each authentication attempt.

File permissions matter. The policy file should be owned by root with mode 0600. It may contain client secrets for confidential OIDC clients. The PAM module runs as root within sshd and can read it.

Issuer Configuration

Each OIDC Identity Provider is configured as an entry in the issuers array. The token's iss claim is matched against these entries. At least one issuer is required.

issuers: - issuer_url: "https://keycloak.example.com/realms/corp" client_id: "prmana" dpop_enforcement: strict # Uncomment for confidential clients: # client_secret: "your-secret-here" # Optional claim transforms: # claim_mapping: # username_claim: preferred_username # transforms: # - type: strip_domain # domain: "example.com"
Copy
Field Required Description
issuer_url Yes Must exactly match the iss claim in tokens (including trailing slash if present)
client_id Yes OAuth client ID; matched against the token's aud claim
client_secret No For confidential clients only
dpop_enforcement No strict (default) | warn | disabled

The issuer URL is used to fetch the OIDC discovery document and JWKS keys. The PAM module fetches JWKS only from the configured URL, never from claims in the token itself.

Username Mapping

By default, Prmana uses the token's preferred_username claim to identify the local Unix account. The claim value must match a local account resolvable via getent passwd (SSSD/NSS).

For IdPs that return non-Unix-friendly usernames, configure a claim transform:

issuers: - issuer_url: "https://login.microsoftonline.com/tenant-id/v2.0" client_id: "prmana" claim_mapping: username_claim: preferred_username transforms: - type: strip_domain domain: "example.com" # Token claim: jdoe@example.com # Mapped to: jdoe
Copy

Groups are always resolved from SSSD/NSS, not from token claims. Your directory (Active Directory, FreeIPA, LDAP) remains the authoritative source for group membership.

ssh_login: # Restrict SSH access to members of these groups (resolved via NSS) login_groups: - developers - sre-team
Copy

Break-Glass Access

Configure a local fallback account for emergency access when the IdP is unavailable. This is mandatory for production deployments.

break_glass: enabled: true accounts: - breakglass alert_on_use: true
Copy

The break-glass account must exist as a local Unix account with a password set:

sudo useradd -m -s /bin/bash breakglass sudo passwd breakglass
Copy

When alert_on_use: true, every break-glass authentication emits a BREAK_GLASS_AUTH audit event at CRITICAL severity. Configure your SIEM to alert on this event immediately.

Before going live: Test that break-glass access works by SSHing as the break-glass account while Prmana is active. The PAM module is configured sufficient -- when it cannot authenticate (IdP down), PAM falls through to pam_unix.so for the break-glass account's local password.

Audit Logging

Prmana logs structured JSON audit events for every authentication attempt -- success, failure, break-glass, and rate-limit events are all recorded.

audit: log_file: "/var/log/prmana-audit.log" syslog_enabled: true
Copy

Create the audit log directory with appropriate permissions:

sudo mkdir -p /var/log/prmana sudo chmod 750 /var/log/prmana sudo chown root:adm /var/log/prmana
Copy

View recent authentication events:

tail -f /var/log/prmana-audit.log | jq . { "event_type": "AUTHENTICATION_SUCCESS", "username": "jdoe", "issuer": "https://keycloak.example.com/realms/corp", "dpop_bound": true, "client_ip": "10.0.1.42", "timestamp": "2026-04-12T10:30:00Z" }
Copy

Events also appear in the system journal and syslog when syslog_enabled: true:

journalctl -u sshd --since "5 minutes ago" | grep prmana
Copy

Hardening

Production deployments should review these settings for their environment.

DPoP enforcement: Set to strict in production. The warn mode accepts bearer tokens without DPoP proof -- use it only during migration from a legacy setup.

security_modes: dpop_required: strict jti_enforcement: strict groups_enforcement: warn
Copy
Setting Recommended Description
dpop_required strict Reject tokens without valid DPoP proof
jti_enforcement strict Reject tokens missing a unique JTI claim
groups_enforcement warn Log when user is not in required groups

Clock skew: The default tolerance is 60 seconds for token staleness and 5 seconds for future-dated proofs. Ensure all servers use NTP. Tighten these values for high-security environments.

JWKS cache TTL: The JWKS cache defaults to 300 seconds (5 minutes). This means the PAM module can continue validating tokens for up to 5 minutes during a transient IdP outage. Increase for environments with unreliable IdP connectivity; decrease for faster key rotation propagation.

File permissions:

sudo chmod 755 /lib/security/pam_prmana.so sudo chown root:root /lib/security/pam_prmana.so sudo chmod 600 /etc/prmana/policy.yaml sudo chown root:root /etc/prmana/policy.yaml sudo chmod 640 /var/log/prmana/audit.log sudo chown root:adm /var/log/prmana/audit.log
Copy

Rate limiting: The PAM module enforces per-user and per-IP rate limits before OIDC processing begins. This protects against authentication flooding without placing load on the IdP.

Host classification: Set host.classification to standard, elevated, or critical to control the default security posture. Critical hosts enforce the strictest defaults and enable additional audit detail.
