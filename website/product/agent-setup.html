Agent Setup

Install and configure the Prmana Agent on your local workstation. The agent handles token acquisition, DPoP proof generation, and credential storage.

Install the Agent

The agent binary is included in the prmana package. If you installed via the package manager on the server, the agent is already available. On your local workstation:

# One-liner install (includes agent) curl -sSf https://get.prmana.com | sh
Copy

Or build from source:

cargo build --release -p prmana-agent sudo cp target/release/prmana-agent /usr/local/bin/
Copy

The agent binary is installed at /usr/local/bin/prmana-agent. Verify it is available:

prmana-agent --version prmana-agent 1.0.0
Copy

Login Flow

Authenticate to your Identity Provider using the device authorization flow. This is a one-time action (tokens are cached and refreshed automatically).

prmana-agent login --issuer https://keycloak.example.com/realms/corp To sign in, open your browser to: https://keycloak.example.com/realms/corp/device Enter code: ABCD-EFGH Waiting for authorization... Login successful. Token cached. DPoP thumbprint: x5t#S256=abc123...
Copy

The agent acquires an access token from the IdP, generates an ephemeral DPoP key pair (if not already provisioned), and stores both securely. Subsequent SSH connections use the cached token automatically.

Token refresh: The agent automatically refreshes tokens before they expire. You do not need to re-run prmana-agent login unless the refresh token itself expires (typically after days or weeks, depending on your IdP configuration).

Signer Backends

The signer backend determines where the DPoP private key is stored. The default software signer works everywhere. Hardware signers provide stronger extraction resistance.

Backend Flag Key Storage Platform
Software (default) --signer software OS keyring or encrypted file All
YubiKey PIV --signer yubikey:9a Non-exportable on device Linux, macOS
TPM 2.0 --signer tpm Non-exportable on chip Linux

Hardware signer support requires feature flags at build time:

# Build with YubiKey support cargo build --release -p prmana-agent --features yubikey # Build with TPM support (Linux only) cargo build --release -p prmana-agent --features tpm # Build with both cargo build --release -p prmana-agent --features yubikey,tpm
Copy

Provision a hardware key before first use:

# Generate a P-256 key on YubiKey slot 9a prmana-agent provision --signer yubikey:9a Provisioning key on yubikey (slot 9a)... Key provisioned successfully on yubikey (slot 9a). DPoP thumbprint: abc123... Run `prmana-agent login --signer yubikey:9a` to authenticate.
Copy
No silent fallback. If a hardware signer is configured but the device is unavailable (YubiKey unplugged, TPM inaccessible), the agent logs an error and refuses to sign. There is no automatic downgrade to a software key. Re-insert the device and re-run login.

Configuration

The agent reads configuration from ~/.config/prmana/config.yaml (user-level) with environment variable overrides prefixed with PRMANA_.

# ~/.config/prmana/config.yaml issuer: "https://keycloak.example.com/realms/corp" client_id: "prmana-agent" # Timeout and clock-skew tuning timeouts: jwks_http_timeout_secs: 10 device_flow_http_timeout_secs: 30 clock_skew_future_secs: 5 clock_skew_staleness_secs: 60 jwks_cache_ttl_secs: 300 ipc_idle_timeout_secs: 60
Copy
Field Default Description
issuer (required) OIDC issuer URL
client_id prmana-agent OAuth client ID registered at the IdP
socket_path auto (UID-derived) Unix socket path for daemon IPC
timeouts.jwks_cache_ttl_secs 300 JWKS cache lifetime in seconds
timeouts.clock_skew_staleness_secs 60 Max age for DPoP proofs and token clock skew

Environment variable overrides use double underscores for nested keys:

PRMANA_ISSUER="https://login.microsoftonline.com/tenant-id/v2.0" prmana-agent login
Copy

Daemon Mode

For production use, run the agent as a background daemon. It listens on a Unix socket for proof requests from SSH clients.

# Start the daemon (foreground, for testing) prmana-agent serve # The socket path is UID-derived by default: # /run/user/<UID>/prmana-agent.sock
Copy

For persistent operation, use the systemd user service:

# Enable and start the agent as a systemd user service systemctl --user enable prmana-agent.socket systemctl --user start prmana-agent.socket # Check status systemctl --user status prmana-agent.socket
Copy

Socket activation means the agent starts on first connection and idles when not in use. The socket permissions are 0600 (owner-only), and the daemon verifies peer credentials on every connection -- only processes running as the same UID can connect.

Credential storage uses the best available backend automatically:

Platform Priority
Linux D-Bus Secret Service → keyutils user keyring → encrypted file
macOS macOS Keychain → encrypted file
File fallback advisory: On CoW filesystems (btrfs, APFS) and SSDs, secure deletion of the file-backed credential store cannot guarantee the original data blocks are overwritten. Prefer a keyring backend, and use full-disk encryption as the primary at-rest protection.
