Core Concepts

How Prmana replaces static SSH keys with OIDC tokens, validated at the host via PAM with cryptographic proof-of-possession.

How Prmana Works

Traditional SSH authentication relies on static key pairs scattered across servers. When someone leaves, finding all their access is archaeology. Enterprise MFA stops at the browser while root access to production requires nothing more than a copied key file.

Prmana bridges this gap by bringing OIDC -- the same protocol behind "Sign in with Google/Microsoft/Okta" -- to Linux PAM. Instead of managing authorized_keys files, your Identity Provider becomes the single source of authentication truth. DPoP token binding (RFC 9449) ensures that even if a token is intercepted, it cannot be used without the client's private key.

User's Machine Identity Provider +-----------------+ +------------------+ +------------------+ | SSH Client |---->| prmana-agent |---HTTPS--->| Okta / Entra / | | | | (token + DPoP) | | Keycloak / Auth0 | +-----------------+ +------------------+ +------------------+ | | SSH with token + DPoP proof v +-----------------+ +------------------+ +------------------+ | sshd |---->| PAM Module |---HTTPS--->| IdP JWKS | | | | (pam_prmana.so) | | (key validation) | +-----------------+ +------------------+ +------------------+

The user authenticates once to their IdP. From that point, SSH access is automatic -- no key files, no key rotation, no scattered authorized_keys. When someone leaves, disable their IdP account and access stops everywhere, immediately.

Token Lifecycle

Every SSH connection follows the same five-step flow. The user sees only the SSH command; everything else happens automatically.

1. IdP issues token. The agent authenticates to the configured OIDC provider via device flow or authorization code + PKCE. The IdP returns an access token. If the IdP supports DPoP, the token includes a cnf.jkt claim binding it to the agent's key pair.

2. Agent acquires and caches. The agent stores the token securely -- in the OS keyring (Secret Service, macOS Keychain, or Linux keyutils) when available, with an encrypted file fallback. Tokens are reused until they expire.

3. DPoP proof generated. For each SSH connection, the agent generates a fresh DPoP proof: a short-lived JWT signed with the ephemeral private key, bound to the target server. Each proof has a unique jti and a recent iat timestamp.

4. PAM validates. On the server, the PAM module validates the token signature against the IdP's published JWKS, checks issuer, audience, and expiration, then verifies the DPoP proof's signature matches the token's cnf.jkt claim. The JTI is recorded to prevent replay.

5. Access granted. The validated OIDC identity is mapped to a local Unix account via SSSD/NSS. If the user exists locally and passes group checks, the session begins. A structured audit event is logged.

Trust Model

Prmana's security rests on clear boundaries between what is trusted, what is verified, and what is rejected.

Trusted:

  • The Identity Provider (Okta, Entra ID, Keycloak, Auth0) -- as a trusted third party for issuing tokens
  • The IdP's JWKS endpoint -- fetched over TLS from the configured issuer URL only, never from token claims
  • The TLS certificate chain -- standard PKI trust for all IdP communication
  • The local OS kernel -- PAM runs as root within sshd; NIST SP 800-63B assumes trusted kernel

Verified on every request:

  • Token signature -- against the IdP's published JWKS keys
  • Token expiration -- rejected if expired, with configurable clock skew tolerance
  • Issuer and audience -- must exactly match the configured values
  • DPoP binding -- proof JWK thumbprint must match the token's cnf.jkt
  • Proof freshness -- iat must be recent, jti must be unique
  • Algorithm -- ES256 only for DPoP proofs; algorithm allowlist for ID tokens
Hard-fail checks -- signature, issuer, audience, expiration, and algorithm enforcement -- can never be disabled. They are not configurable. A failure in any of these always rejects the token.

Username Mapping

Prmana maps OIDC identities to local Unix accounts through your existing directory infrastructure, not through token claims alone.

The token's preferred_username claim identifies the user. This claim is compared against local Unix accounts resolved via SSSD/NSS. The user must exist as a local account -- Prmana authenticates identity but does not create accounts.

# Verify the user exists locally via SSSD getent passwd jdoe jdoe:*:10042:10042:Jane Doe:/home/jdoe:/bin/bash

Groups are resolved from SSSD/NSS, not from OIDC token claims. Your directory (FreeIPA, Active Directory, LDAP) remains the authoritative source for group membership and local authorization. This prevents split-brain authorization where IdP group claims disagree with local directory state.

Claim transforms are available for IdPs where preferred_username includes a domain suffix. For example, Azure AD may return jdoe@example.com -- configure a strip_domain transform to map it to the local account jdoe.

Break-Glass Access

Never deploy OIDC authentication as the only authentication path. If your IdP goes down, you must still be able to access your servers.

Prmana is configured as sufficient in the PAM stack, with pam_unix.so as a fallback. When OIDC authentication fails -- IdP outage, network failure, misconfiguration -- PAM falls through to local password authentication for designated break-glass accounts.

# Create break-glass account before enabling Prmana sudo useradd -m -s /bin/bash breakglass sudo passwd breakglass # Store the password in your organization's secure vault

Break-glass usage always emits a BREAK_GLASS_AUTH audit event at CRITICAL severity. Configure your SIEM to alert on this event immediately -- break-glass use should be rare and investigated.

Pre-deployment requirement: Configure and test break-glass access before going live. Include break-glass testing in quarterly DR exercises. Getting locked out of servers because your IdP is down is a catastrophic, career-affecting failure mode.
