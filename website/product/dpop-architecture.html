DPoP Architecture

How Prmana uses RFC 9449 Demonstrating Proof-of-Possession to bind tokens to a client key pair, making stolen tokens unusable.

Why Not Bearer Tokens?

Standard OAuth access tokens are bearer tokens. Anyone who has the token can use it. If an attacker intercepts a bearer token from a log file, a network capture, or a compromised intermediary, they can replay it from any machine until it expires.

For SSH authentication, this risk is severe. A bearer token grants shell access. Lateral movement becomes trivial.

Scenario Bearer Token DPoP-Bound Token
Token intercepted in transit Attacker has full access Useless without private key
Token found in log file Attacker has full access Useless without private key
Token expires Minutes to hours Minutes to hours
Centralized revocation Yes (IdP action) Yes (IdP action)

DPoP does not eliminate all credential theft risks. If an attacker can extract both the token and the private key from a compromised client, they can act as that client until the token expires. But DPoP dramatically reduces the blast radius: a stolen token alone is worthless.

RFC 9449 in 60 Seconds

DPoP adds three steps to the standard OAuth flow:

1. The client (prmana-agent) generates an ephemeral P-256 ECDSA key pair at startup.

2. When requesting a token from the IdP, the client includes a DPoP proof -- a JWT signed with the private key, containing the public key in the header. The IdP binds the issued token to that key by embedding the key's thumbprint in the token's cnf.jkt claim.

3. On every subsequent use, the client sends a fresh DPoP proof alongside the token. The server verifies that the proof was signed by the key matching the token's thumbprint.

Token Request: Client --> IdP: Authorization + DPoP Proof (containing public key) IdP --> Client: Access token with cnf: { jkt: "thumbprint" } Token Use (every SSH connection): Client --> Server: Access token + fresh DPoP Proof (signed) Server: Verify proof signature matches token's jkt claim
Algorithm choice: ES256 (P-256 ECDSA). NIST-approved, 128-bit security level, fast signing/verification, compact 64-byte signatures. DPoP proofs in Prmana always use ES256. The alg: "none" and symmetric algorithms are rejected unconditionally.

JWK Thumbprint

The binding between a token and its key is a JWK Thumbprint (RFC 7638). The thumbprint is computed by hashing the canonical JSON representation of the public key:

1. Construct canonical JSON (RFC 7638 member ordering) {"crv":"P-256","kty":"EC","x":"<base64url>","y":"<base64url>"} 2. SHA-256 hash the canonical JSON 3. Base64url-encode the hash = thumbprint

Prmana uses hardcoded canonical values "EC" and "P-256" for kty and crv in the thumbprint computation, ignoring user-supplied values. This prevents an attacker from supplying non-canonical values to produce a different thumbprint while using a valid key. All thumbprint comparisons use constant-time operations to prevent timing attacks.

Proof Structure

A DPoP proof is a JWT with specific claims that bind it to a single request:

DPoP Proof Header: { "typ": "dpop+jwt", "alg": "ES256", "jwk": { "kty": "EC", "crv": "P-256", "x": "<base64url-encoded x coordinate>", "y": "<base64url-encoded y coordinate>" } } DPoP Proof Payload: { "jti": "unique-id-per-proof", "htm": "SSH", "htu": "server.example.com", "iat": 1713000000, "exp": 1713000060 }
Claim Purpose Validation
jti Unique proof identifier Must not have been seen before (replay protection)
htm HTTP method binding Must match the request method (SSH for Prmana)
htu Target URI binding Must match the target server hostname
iat Issued-at timestamp Must be recent (within max proof age, default 60s)

Replay Protection

Even with DPoP binding, an attacker who captures both the token and a valid proof could attempt to replay them. Prmana uses three layers of defense:

1. JTI uniqueness. Every DPoP proof carries a unique jti claim. The PAM module records seen JTI values and rejects duplicates. The JTI store uses filesystem-based atomic operations, making it safe across forked sshd worker processes -- a critical property for real PAM deployments.

2. Server-issued nonce. The server can issue a nonce that the client must include in the next proof. Nonces are single-use with no TOCTOU window (atomic removal from cache). Each nonce is 256 bits from a CSPRNG.

3. Proof freshness. The iat claim is checked against a configurable maximum proof age (default 60 seconds) and clock skew tolerance (default 5 seconds into the future). Old proofs are rejected even if their JTI has not been seen.

Cross-process safety. PAM runs inside sshd, which may fork worker processes. In-memory-only replay caches would miss replays sent to sibling workers. Prmana's filesystem-backed JTI store ensures replay detection works across all sshd processes on the host.

Hardware Key Binding

Software DPoP keys are stored encrypted in the OS keyring or a file fallback. This protects against casual theft but not against a determined attacker with root access to the client machine. For higher assurance, Prmana supports a progression of hardware-backed signers:

Backend Key Storage Extraction Resistance
Software (default) OS keyring / encrypted file Protected by OS permissions; root can extract
YubiKey PIV On-device, non-exportable Key never leaves the hardware; requires physical touch
TPM 2.0 On-chip, non-exportable Key bound to physical TPM; survives OS compromise

With hardware-backed keys, the DPoP private key material is never accessible to software. Even a root-level attacker cannot extract it. The agent acts as a signing oracle -- it can request signatures from the hardware, but cannot export the key itself.

# Provision a YubiKey-backed DPoP key prmana-agent provision --signer yubikey:9a # Or a TPM-backed key (Linux only) prmana-agent provision --signer tpm
